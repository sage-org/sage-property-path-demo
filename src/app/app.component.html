SaGe-Path: Pay-as-you-go SPARQL Property Path Queries Processing using Web Preemption

University of Nantes, LS2N

Julien Aimonier-Davat (julien.aimonier-davat@univ-nantes.fr) Hala Skaf-Molli (hala.skaf@univ-nantes.fr) Pascal Molli (pascal.molli@univ-nantes.fr)

1. Introduction

SPARQL property path queries represent an important tool to write complex navigational queries over knowledge graphs. However the evaluation of property path queries on online knowledge graphs are often interrupted by fair use policies, preventing them to return complete results [1]. To overcome this issue, state-of-art approaches propose to decompose SPARQL property path queries into a set of subqueries that are guaranteed to terminate. For instance, the Triple Pattern Fragment model [3] ensures the termination of any triple pattern query, while the Web preemption model [2] ensures the termination of any BGP query. However, such a decomposition can generate a high number of subqueries and require to transfer a lot of intermediate results from the server to the client [1].

2. SaGe-Path

SaGe-Path is a new approach to compute SPARQL property path queries on online knowledge graphs with guaranteed complete results. Based on the web preemption, SaGe-Path proposes a new kind of collaboration between the client and the server. If web preemption cannot compute a transitive closure on server-side, in our ESWC2020 article [1], we demonstrated that a preemptable server can fairly process a Partial Transitive Closure (PTC). As a transitive closure, a PTC is computed using a graph traversal algorithm except that the exploration of the graph is limited to a predefined depth k. Nodes visited at depth k are called frontier nodes. To ensure complete results, the server relies on the client by sending it the frontier nodes. The SaGe-Path client is then able to generate new path queries that restart the exploration from these frontier nodes.

Compared to the state-of-art, all transitive closures and BGPs are now executed on the preemptable server, only results and frontier nodes are shipped to the client. As no more joins are performed on the client, performance drastically improved.

3. Demonstration overview

In this demonstration, we offer the possibility for the user to understand how property path queries are evaluated using SaGe-Path and what is the impact of the different parameters on performance. For example, the user can change the time quantum and the depth limit used by the PTC operator. In a real world configuration, these two parameters are defined by the data provider. Using our interface, the user can write and evaluate its own path query. Different views allow the user to observe the frontier nodes sent by the server and how the SaGe-Path client expand these nodes by generating new path queries. Real time statistics are also available to observe the impact of the different parameters on performance and to understand the overheads of the approach.

A video that explain how to use the user interface is available on YouTube. In this video, we show you how a SPARQL property path query that does not terminate on Wikidata, is evaluated with SaGe-Path. The implementation of this Web client is available on Github.
